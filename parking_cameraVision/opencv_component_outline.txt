Parking Detector (OpenCV) — Component Overview
==================================================

Purpose
-------
This component is the "vision + detection" part of the parking project.

It:
1) Reads frames from a camera, video file, single image, or a folder of images
2) Uses pre-defined parking spot regions (polygons) to decide which spots are OCCUPIED vs FREE
3) Produces a clean JSON payload (spotId + occupied + confidence) for the backend/web team
4) Optionally overlays the result on the video/image (red/green fill) for demo/debugging
5) Optionally POSTs the JSON to a backend endpoint (when available)

Why we use polygon ROIs (and not fully auto line detection)
-----------------------------------------------------------
Auto-detecting painted parking lines reliably is hard (shadows, glare, worn paint, perspective).
For a mounted camera (especially angled), a robust approach is:

- "Calibrate once": manually define each parking stall boundary as a polygon ROI
- "Detect continuously": compute occupancy inside each ROI on each update interval

This works for both angled and more top-down views because the polygon can match perspective.

High-level Data Flow
--------------------
Camera / Program  ->  Backend Server  ->  Website UI

This component ONLY handles the left side:
- It outputs occupancy data (JSON) that the backend can store and the website can display.

Key Modes
---------
1) LABEL MODE (one-time per camera angle)
   Goal: Create spots.json by clicking stall polygons.
   - User loads a reference frame (first frame from camera/video or a still image)
   - User clicks points around a stall (3+ points)
   - Press 'n' to finalize that stall
   - Repeat for all stalls
   - Press 's' to save to spots.json

   Output: spots.json (calibration file)
   Example structure:
   {
     "cameraId": "lot-1",
     "spots": [
       {"id": "S1", "polygon": [[x,y],[x,y],[x,y],[x,y]]},
       {"id": "S2", "polygon": [[x,y],[x,y],[x,y],[x,y]]}
     ]
   }

2) DETECT MODE (continuous)
   Goal: Read frames and classify each spot as occupied/free.
   Inputs:
   - a live webcam index (e.g., 0)
   - a video file path
   - a single image path
   - a folder of images (for testing before cameras are ready)
   - spots.json (from label mode)

   Outputs:
   - prints JSON payload to stdout
   - writes JSON payload to a file (status.json by default)
   - optionally POSTs payload to backend (if --post is provided)
   - optionally shows overlay window (if --show is provided)

Core Detection Approach (Easy + Robust MVP)
-------------------------------------------
We use "edge density" inside each spot ROI:

Per frame:
1) Convert frame to a consistent width (important so ROI coordinates match)
2) Convert to grayscale
3) Blur slightly to reduce noise
4) Run Canny edge detection
5) For each spot polygon:
   a) Create a mask for the polygon
   b) Count edge pixels inside the polygon
   c) Compute edge_density = edge_pixels / polygon_area
   d) Convert edge_density into OCCUPIED/FREE using thresholds and smoothing

Why edge density works (most of the time):
- A parked car adds edges/texture (windows, tires, reflections) compared to empty asphalt.

Stabilizing the Occupancy Result (prevents flicker)
---------------------------------------------------
Real camera feeds are noisy. To stop rapid flipping between FREE and OCCUPIED, we use:

1) Hysteresis thresholds
   - FREE -> OCCUPIED when score >= ON threshold
   - OCCUPIED -> FREE only when score < OFF threshold
   (OFF threshold is lower than ON threshold)

2) Temporal smoothing (majority vote)
   - Keep the last N results per spot
   - Decide by majority vote across that window

Confidence Score (simple heuristic)
-----------------------------------
The program outputs a confidence value in [0..1].
It is a heuristic based on how far the score is from the decision boundary.
It can be improved later (e.g., with ML or background subtraction).

Overlay / Visualization (for demo + debugging)
----------------------------------------------
When --show is enabled, the program:
- Draws each spot polygon outline
- Fills it GREEN if free, RED if occupied
- Writes a label like: "S1:OCC" or "S1:FREE"

This is helpful to:
- confirm ROIs are correct
- tune thresholds
- show results live during a demo

JSON Output Contract (what backend/web consumes)
------------------------------------------------
Each update emits a payload like:

{
  "cameraId": "lot-1",
  "timestamp": "2026-03-04T11:12:30-07:00",
  "spots": [
    {"id":"S1","occupied":true,"confidence":0.87},
    {"id":"S2","occupied":false,"confidence":0.74}
  ]
}

This component does NOT decide how the website looks.
It only provides a reliable, simple status feed.

Command Examples
----------------
Label (from webcam):
  python3 parking_detector.py label --source 0 --spots spots.json

Label (from image):
  python3 parking_detector.py label --source reference.jpg --spots spots.json

Detect (from webcam + overlay + write status.json):
  python3 parking_detector.py detect --source 0 --spots spots.json --show --out status.json

Detect (from folder of test images):
  python3 parking_detector.py detect --source ./images --spots spots.json --show

Detect + POST to backend endpoint:
  python3 parking_detector.py detect --source 0 --spots spots.json --post http://localhost:3000/api/spots/update

Tuning Notes (common adjustments)
---------------------------------
If occupancy is wrong:
- Increase ON/OFF thresholds if it marks too many spots occupied
- Decrease thresholds if it misses cars
- Increase history length to reduce flicker
- Ensure the camera is stable and the frame width is consistent

Future Enhancements (optional)
------------------------------
- Add background subtraction (MOG2/KNN) for mounted cameras with stable scenes
- Hybrid scoring: combine edge density + foreground ratio
- Calibrate empty-lot baselines per ROI
- ML classifier for highest accuracy
